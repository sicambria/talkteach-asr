# Internationalizing the UI (#36)

TalkTeach's whole point is that a child can use it without help — but the
plain-language strings ("Let's record!", "Press to talk") are **English-only**.
A non-English-speaking child can't read them, which quietly excludes exactly the
low-resource-language users the wav2vec2/XLS-R engine (#26) is meant to serve. So
the UI strings themselves need translating, not just the model.

## The catalog approach (`ui/src/lib/i18n.js`)

A dependency-free Svelte-store catalog, deliberately tiny:

```js
import { locale, t, availableLocales } from './lib/i18n.js';
// in a component:  {$t('record.title')}
```

- `CATALOGS` maps a **language code → { key: string }**. Add a language by adding
  its map — nothing else changes.
- `t` is a derived store: `$t('key')` looks up the active `locale`, **falls back to
  English, then to the key itself**, so a partial translation never blanks the UI
  or crashes a screen.
- `locale` is a writable store; a language picker (or the project's language, #29)
  sets it.

## How strings get keys

Keys are namespaced by screen: `record.*`, `check.*`, `teach.*`, `try.*`, plus
`common.*` for shared chrome. The migration is **incremental**: each hard-coded
string in a `Screen*.svelte` becomes `{$t('screen.key')}` and gains an entry under
`en`. Do it screen by screen so the app stays shippable throughout — an
un-migrated string just stays English.

## Pairs with the TypeScript migration (D-011)

The UI is deliberately plain-JS Svelte 4 today (`svelte-check` runs with
`checkJs:false`, D-011). A full i18n pass — typed keys, a checked catalog, maybe
ICU plurals — is best done alongside the planned **Svelte 5 / TypeScript
migration**, where the catalog can be a typed `Record<Locale, Messages>` and
missing-key errors become compile-time. Until then the JS scaffold is enough to
start keying strings.

## What's pending

- Migrate the remaining hard-coded strings across the five screens to `$t(...)`.
- Add at least one non-English catalog end-to-end as the proof case.
- A language picker wired to `locale` (and ideally to the project language).
- Right-to-left handling and font coverage for non-Latin scripts (ties to
  `docs/ACCESSIBILITY.md`).

## Status

**Tier B** (#36). The `i18n.js` catalog, the `t`/`locale` stores, and the
English base catalog exist and are usable; most screen strings are not yet keyed,
and no second language ships yet. The full typed pass rides with the D-011 TS
migration.
