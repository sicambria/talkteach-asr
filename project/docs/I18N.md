# Internationalizing the UI (#36)

> **Scope:** this doc is about the **interface language** — the app's own buttons
> and text. For the **speech languages the model can learn** (~99 Whisper + XLS-R
> for any other), see [`LANGUAGES.md`](LANGUAGES.md). The two are independent: the
> UI can be English while you train a Hungarian or Swahili model.

TalkTeach's whole point is that a child can use it without help — but the
plain-language strings ("Let's record!", "Press to talk") are **English-only**.
A non-English-speaking child can't read them, which quietly excludes exactly the
low-resource-language users the wav2vec2/XLS-R engine (#26) is meant to serve. So
the UI strings themselves need translating, not just the model.

## The catalog approach (`ui/src/lib/i18n.js`)

A dependency-free Svelte-store catalog, deliberately tiny:

```js
import { locale, t, availableLocales } from './lib/i18n.js';
// in a component:  {$t('record.title')}
```

- `CATALOGS` maps a **language code → { key: string }**. Add a language by adding
  its map — nothing else changes.
- `t` is a derived store: `$t('key')` looks up the active `locale`, **falls back to
  English, then to the key itself**, so a partial translation never blanks the UI
  or crashes a screen.
- `locale` is a writable store; a language picker (or the project's language, #29)
  sets it.

## How strings get keys

Keys are namespaced by screen: `record.*`, `check.*`, `teach.*`, `try.*`, plus
`common.*` for shared chrome. The migration is **incremental**: each hard-coded
string in a `Screen*.svelte` becomes `{$t('screen.key')}` and gains an entry under
`en`. Do it screen by screen so the app stays shippable throughout — an
un-migrated string just stays English.

## Pairs with the TypeScript migration (D-011)

The UI is deliberately plain-JS Svelte 4 today (`svelte-check` runs with
`checkJs:false`, D-011). A full i18n pass — typed keys, a checked catalog, maybe
ICU plurals — is best done alongside the planned **Svelte 5 / TypeScript
migration**, where the catalog can be a typed `Record<Locale, Messages>` and
missing-key errors become compile-time. Until then the JS scaffold is enough to
start keying strings.

## What shipped (the plumbing, #36)

- **Every static screen string is keyed.** Screen0–4 plus the new pre-flight
  screen render their chrome through `$t('...')`; the `en` catalog in `i18n.js`
  holds all the keys (namespaced `newproject.*`, `preflight.*`, `record.*`,
  `check.*`, `teach.*`, `try.*`, `common.*`, `app.*`).
- **A language switcher** in the topbar, bound to the `locale` store and populated
  from `availableLocales()`.
- **A synthetic `qa` pseudo-locale**, generated programmatically from `en` (every
  value accented + bracketed, `Hello → ⟦Ħéĺĺó⟧`). It is *not* a shipped
  translation — it exists so the switcher visibly swaps the whole UI through
  `t()` (proving the plumbing) and so any un-keyed string is easy to spot (it
  stays plain English). Zero-maintenance: it tracks `en` automatically.

## What's still pending

- **Real translations (l10n).** Only `en` is a real catalog; adding a genuine
  second language needs translators. A partial catalog is safe — missing keys fall
  back to English.
- **Typed keys / checked catalog.** Rides with the D-011 Svelte 5 / TS migration.
- **Wire `locale` to the project language** (#29) so the UI can follow the chosen
  speech language automatically.
- **Right-to-left handling** and font coverage for non-Latin scripts (ties to
  `ACCESSIBILITY.md`).
- **The grown-up Arena (Screen5)** and JS-side error-fallback strings are left in
  English for now — the technical/rare surfaces, not the child-facing flow.

## Status

**Tier B → plumbing delivered** (#36). The `i18n.js` catalogs, the `t`/`locale`
stores, the keyed screens, the switcher, and the pseudo-locale all ship and are
verified. Real second-language translations and the full typed pass remain,
riding with the D-011 TS migration.
